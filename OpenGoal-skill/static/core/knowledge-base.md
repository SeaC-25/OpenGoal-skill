# DDD/SOA 知识库 — OpenGoal Skill

本文档是 OpenGoal skill 的专业知识库，包含领域驱动设计（DDD）和面向服务架构（SOA）的核心概念、识别方法和最佳实践。

---

## 一、领域驱动设计（DDD）

### 1.1 领域（Domain）识别

**定义**：业务问题所在的领域，如"电商"、"医疗"、"金融"。

**识别方法**：
- 关键词识别：订单、支付、商品 → 电商领域
- 语义分析：病人、诊断、处方 → 医疗领域
- 上下文推断：如果用户提到"库存"、"购物车"、"物流" → 电商领域

**输出格式**：
```
识别的领域：电商领域
核心业务：在线销售商品，管理订单、支付、物流
```

### 1.2 限界上下文（Bounded Context）

**定义**：领域的边界，一个限界上下文内的术语和模型是一致的。

**识别方法**：
- 按业务能力划分：订单上下文、支付上下文、库存上下文
- 按数据边界划分：每个上下文有自己的数据库
- 按团队边界划分：不同团队负责不同上下文

**示例**：
```
电商系统的限界上下文：
1. 商品目录上下文（Product Catalog Context）
   - 职责：商品信息管理、分类、搜索
   - 核心实体：商品、分类、品牌
   
2. 交易上下文（Trading Context）
   - 职责：订单管理、支付处理
   - 核心实体：订单、订单项、支付记录
   
3. 履约上下文（Fulfillment Context）
   - 职责：库存管理、物流配送
   - 核心实体：库存、发货单、物流信息
```

### 1.3 实体（Entity）与值对象（Value Object）

**实体**：有唯一标识的对象，生命周期中标识不变。
- 示例：用户（通过userId标识）、订单（通过orderId标识）

**值对象**：无唯一标识，只有属性，可替换。
- 示例：地址（相同的地址可以有多个实例）、价格、日期范围

**识别规则**：
- 需要追踪和查询 → 实体
- 只关心属性值 → 值对象
- 可以被另一个相同的对象替换 → 值对象

**示例**：
```
订单实体（Entity）：
- orderId: String（唯一标识）
- userId: String
- shippingAddress: Address（值对象）
- totalPrice: Money（值对象）
- status: OrderStatus
- createdAt: DateTime

地址值对象（Value Object）：
- province: String
- city: String
- street: String
- zipCode: String
```

### 1.4 聚合根（Aggregate Root）

**定义**：聚合的入口实体，外部只能通过聚合根访问聚合内的对象。

**识别方法**：
- 找到业务的"边界"：订单是一个聚合，包含订单项、支付信息
- 事务边界：一次业务操作修改的对象集合
- 不变量保护：需要维护的业务规则

**示例**：
```
订单聚合（Order Aggregate）：
- 聚合根：Order
- 内部对象：OrderItem（订单项）、PaymentInfo（支付信息）
- 业务规则：订单总价 = Σ(订单项单价 × 数量)
- 外部访问：只能通过 Order.addItem()、Order.pay() 等方法

用户聚合（User Aggregate）：
- 聚合根：User
- 内部对象：UserProfile（用户资料）、LoginCredential（登录凭证）
- 业务规则：密码必须加密存储
- 外部访问：只能通过 User.updateProfile()、User.changePassword() 等方法
```

### 1.5 领域事件（Domain Event）

**定义**：领域中发生的、业务关心的事实记录，过去时态命名。

**识别方法**：
- 关键词："当...时"、"...之后"、"通知"、"触发"
- 状态变化：实体的重要状态改变
- 业务里程碑：注册完成、支付成功、发货等

**事件链设计**：
对于复杂流程，建议完整事件链。

**示例**：
```
用户注册流程的事件链：
1. UserRegistrationStarted（用户注册开始）
   - 触发条件：用户点击注册按钮
   - 数据：userId, email, registrationTime
   
2. EmailVerificationSent（邮箱验证邮件已发送）
   - 触发条件：注册信息验证通过
   - 数据：userId, email, verificationToken
   
3. UserRegistered（用户注册完成）
   - 触发条件：邮箱验证通过
   - 数据：userId, email, registeredTime
```

**事件驱动架构的优势**：
- 松耦合：服务间通过事件通信，不直接依赖
- 可扩展：新增监听者不影响发布者
- 异步处理：提高系统响应速度
- 事件溯源：可回溯历史状态

### 1.6 领域服务（Domain Service）

**定义**：不属于任何实体的业务逻辑，操作多个聚合或外部资源。

**识别方法**：
- 跨聚合的业务逻辑
- 需要多个实体协作的操作
- 无状态的业务操作

**示例**：
```
库存分配服务（InventoryAllocationService）：
- 职责：为订单分配库存，涉及订单和库存两个聚合
- 方法：allocateInventory(orderId, productId, quantity)
- 业务规则：
  - 检查库存是否充足
  - 锁定库存
  - 更新订单状态

转账服务（TransferService）：
- 职责：在两个账户间转账
- 方法：transfer(fromAccountId, toAccountId, amount)
- 业务规则：
  - 检查余额是否充足
  - 扣减源账户
  - 增加目标账户
  - 记录转账记录
```

### 1.7 仓储（Repository）模式

**定义**：数据访问的抽象层，隐藏数据存储细节。

**何时需要**：
- 需要支持多种数据源（MySQL、MongoDB、缓存）
- 需要将领域模型与数据访问解耦
- 需要复杂查询封装

**示例**：
```
UserRepository 接口：
- findById(userId: String): User
- findByEmail(email: String): User
- save(user: User): void
- delete(userId: String): void

实现可以是：
- UserRepositoryMySQL（MySQL实现）
- UserRepositoryMongo（MongoDB实现）
- UserRepositoryCached（带缓存的实现）
```

### 1.8 业务规则提取与表达

**规则类型**：
1. **验证规则**：密码必须8位以上、邮箱格式校验
2. **计算规则**：总价 = 单价 × 数量 - 折扣
3. **派生规则**：VIP用户 = 消费满1000元的用户
4. **约束规则**：库存不能为负数、订单不可重复提交

**表达方式**：
使用结构化描述（规范模式 Specification Pattern）：

```
业务规则：VIP折扣规则
触发条件：用户类型为VIP
排除条件：商品类别为图书
执行动作：价格 × 0.8
优先级：P1
验证方式：测试用例覆盖所有场景

业务规则：库存不足拦截
触发条件：下单时库存 < 购买数量
执行动作：拒绝订单，提示"库存不足"
优先级：P0
验证方式：集成测试模拟库存不足场景
```

## 二、面向服务架构（SOA）

### 2.1 服务边界识别

**定义**：明确每个服务的职责范围，避免职责重叠或不清晰。

**识别原则**：
- 单一职责：一个服务只负责一个业务能力
- 高内聚：服务内部功能紧密相关
- 低耦合：服务间依赖尽可能少

**示例**：
```
用户服务（UserService）：
- 职责：用户注册、登录、资料管理
- 不包含：订单管理、支付处理

订单服务（OrderService）：
- 职责：订单创建、查询、状态管理
- 不包含：支付处理、库存扣减（通过事件通知）
```

### 2.2 服务拆分判断标准

**何时拆分为多个服务**：
1. **不同业务域**：用户管理 vs 订单管理 → 拆分
2. **不同技术栈需求**：Python机器学习模块 vs Java业务模块 → 拆分
3. **不同扩展需求**：高并发的搜索服务 vs 低频的后台管理 → 拆分
4. **不同团队负责**：团队A负责用户模块，团队B负责订单模块 → 拆分
5. **数据边界清晰**：不共享数据库 → 拆分

**何时保持单一服务**：
1. **功能内聚**：用户注册、登录、资料管理都属于用户管理 → 单一服务
2. **团队规模小**：< 10人的团队 → 单体应用优先
3. **变更频率一致**：功能同步迭代 → 单一服务
4. **事务需求强**：需要强一致性保证 → 单一服务（或使用Saga）

**拆分原则（按功能拆分）**：
```
电商系统服务拆分：
1. 用户服务（UserService）
   - 用户注册、登录、资料管理
   
2. 商品服务（ProductService）
   - 商品信息管理、分类、搜索
   
3. 订单服务（OrderService）
   - 订单创建、查询、状态管理
   
4. 支付服务（PaymentService）
   - 支付处理、退款
   
5. 物流服务（LogisticsService）
   - 发货、物流跟踪
```

### 2.3 服务编排（Orchestration）vs 服务协作（Choreography）

**服务编排（Orchestration）**：
- **定义**：中心化控制，有一个"指挥者"（Orchestrator）协调所有服务
- **适用场景**：
  - 流程固定，步骤明确
  - 需要事务保证（Saga模式）
  - 需要全局视图和控制
- **示例**：订单服务作为编排者，依次调用支付、库存、物流服务

**优点**：
- 流程清晰，易于理解和调试
- 集中式错误处理和补偿
- 全局视图，便于监控

**缺点**：
- 编排者成为单点故障和性能瓶颈
- 耦合度较高
- 扩展性受限

**服务协作（Choreography）**：
- **定义**：去中心化，服务通过事件自主协作，无中央协调者
- **适用场景**：
  - 松耦合，高扩展性
  - 服务自治性强
  - 不需要强事务保证
- **示例**：订单服务发布 `OrderPlaced` 事件，支付、库存、物流服务各自监听并处理

**优点**：
- 松耦合，高扩展性
- 无单点故障
- 服务自治，独立演进

**缺点**：
- 流程分散，难以理解全局
- 调试和监控困难
- 事件顺序和一致性难保证

**混合模式（推荐）**：
- 关键操作（如库存扣减、支付）使用编排模式保证一致性
- 非关键操作（如邮件通知、积分增加）使用协作模式解耦

**示例**：
```
用户下单后的处理（混合模式）：

编排部分（关键路径）：
OrderService 作为编排者：
1. 调用 PaymentService.processPayment()
2. 调用 InventoryService.deductStock()
3. 如果失败，执行补偿（退款、释放库存）

协作部分（非关键路径）：
OrderService 发布 OrderPlaced 事件：
- EmailService 监听 → 发送确认邮件
- PointsService 监听 → 增加用户积分
- AnalyticsService 监听 → 记录分析数据
```

### 2.4 API 契约设计

**Contract-First 原则**：
- 先定义 API 契约（OpenAPI/Swagger），再实现
- 前后端并行开发，减少集成问题

**API 版本管理策略**：
1. **URI版本**：`/api/v1/users`、`/api/v2/users`（推荐，默认方案）
2. **Header版本**：`Accept: application/vnd.api.v2+json`
3. **Query参数版本**：`/api/users?version=2`（不推荐）

**版本化规则**：
- 所有 API 默认版本化（避免后期难以引入版本管理）
- 破坏性变更必须升级主版本
- 非破坏性变更可在同一版本内扩展

**示例**：
```
OpenAPI 契约示例：
POST /api/v1/users
Request:
{
  "email": "user@example.com",
  "password": "securePassword123"
}
Response (201 Created):
{
  "userId": "uuid-1234",
  "email": "user@example.com",
  "createdAt": "2024-07-06T12:00:00Z"
}
Response (400 Bad Request):
{
  "error": "INVALID_EMAIL",
  "message": "邮箱格式不正确"
}
```

### 2.5 服务依赖管理

**依赖类型**：
1. **同步依赖**：服务A直接调用服务B的API
2. **异步依赖**：服务A发布事件，服务B监听
3. **数据依赖**：服务A需要服务B的数据

**循环依赖检测与解决**：
- **检测**：自动识别 A→B→C→A 的循环依赖
- **解决方案**：
  1. 引入中介者服务
  2. 合并服务（如果职责重叠）
  3. 反转依赖方向
  4. 使用事件解耦

**示例**：
```
循环依赖问题：
- 订单服务依赖用户服务（获取用户信息）
- 用户服务依赖订单服务（显示用户订单数量）

解决方案1：引入BFF（Backend for Frontend）
- OrderService 和 UserService 都不直接依赖对方
- 前端调用 BFF，BFF 聚合两个服务的数据

解决方案2：数据冗余
- UserService 缓存用户的订单数量（通过监听 OrderPlaced 事件更新）
```

### 2.6 服务可观测性

**三大支柱**：
1. **日志（Logging）**：记录关键操作和错误
2. **监控（Monitoring）**：实时监控性能指标（QPS、RT、错误率）
3. **追踪（Tracing）**：分布式调用链追踪（如OpenTelemetry）

**在需求分析中的体现**：
- 识别到分布式系统 → 必须包含可观测性需求
- 验收标准中包含日志完整性检查

### 2.7 Saga 模式（分布式事务）

**定义**：通过一系列本地事务和补偿事务实现分布式事务。

**何时需要**：
- 跨服务的事务一致性需求
- 如：订单-支付-库存-物流 的完整流程需要保证一致性

**两种实现**：
1. **编排式Saga**：中央协调者管理事务流程
2. **协作式Saga**：服务通过事件自主协作

**示例**：
```
订单流程的 Saga（编排式）：

正向流程：
1. OrderService.createOrder() → 订单已创建
2. PaymentService.processPayment() → 支付成功
3. InventoryService.deductStock() → 库存已扣减
4. LogisticsService.arrangeShipping() → 已安排发货

补偿流程（如果第3步失败）：
1. PaymentService.refund() → 退款
2. OrderService.cancelOrder() → 取消订单
```

**在需求分析中的体现**：
- 自动识别跨服务事务场景
- 建议使用 Saga 模式
- 在验收标准中包含补偿逻辑测试

## 三、架构模式决策树

### 何时选择同步调用（编排）
- ✅ 需要立即响应
- ✅ 流程固定，步骤明确
- ✅ 强一致性要求
- ✅ 调用链短（< 3层）

### 何时选择异步事件（协作）
- ✅ 允许最终一致性
- ✅ 需要高扩展性
- ✅ 服务自治性强
- ✅ 调用链长或复杂

### 何时选择混合模式（推荐）
- ✅ 关键操作同步（如支付、库存扣减）
- ✅ 非关键操作异步（如邮件、积分、通知）
- ✅ 兼顾一致性和扩展性

### DDD 核心概念速查

| 概念 | 定义 | 识别方法 | 示例 |
|------|------|----------|------|
| 领域 | 业务问题领域 | 关键词、语义分析 | 电商、医疗 |
| 限界上下文 | 领域边界 | 按业务能力划分 | 订单上下文、支付上下文 |
| 实体 | 有唯一标识的对象 | 需要追踪和查询 | 用户、订单 |
| 值对象 | 只有属性的对象 | 可替换、无标识 | 地址、价格 |
| 聚合根 | 聚合的入口 | 事务边界、业务规则 | 订单（含订单项） |
| 领域事件 | 业务事实记录 | 关键词、状态变化 | OrderPlaced |
| 领域服务 | 跨聚合业务逻辑 | 无状态、多实体协作 | 库存分配服务 |
| 仓储 | 数据访问抽象 | 需要多数据源 | UserRepository |

### SOA 核心概念速查

| 概念 | 定义 | 何时使用 | 示例 |
|------|------|----------|------|
| 服务拆分 | 按功能拆分服务 | 不同业务域、技术栈 | 用户服务、订单服务 |
| 编排 | 中心化控制 | 流程固定、强一致性 | 订单服务调用支付、库存 |
| 协作 | 去中心化事件驱动 | 松耦合、高扩展性 | OrderPlaced事件通知多个服务 |
| API契约 | 接口定义 | 前后端并行开发 | OpenAPI/Swagger |
| Saga | 分布式事务 | 跨服务事务一致性 | 订单-支付-库存-物流 |
