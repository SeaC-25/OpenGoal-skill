# 核心工作契约 — OpenGoal Skill

本文档定义 OpenGoal skill 的核心工作契约和质量标准。

---

## 工作原则

### 1. 需求优先，技术次之
- 先理解业务需求，再选择技术方案
- 避免"技术驱动"，应该是"需求驱动"
- 当技术选型存在多种可能时，优先选择最适合业务场景的方案

### 2. 默认异步、松耦合
- 优先采用事件驱动架构
- 服务间通过消息队列解耦
- 关键操作（如库存扣减、支付）可使用编排模式保证一致性

### 3. 显式优于隐式
- 所有假设必须标注
- 所有约束必须明确
- 所有风险必须提示

### 4. 质量不妥协
- 安全是第一优先级（密码加密、SQL注入防护等）
- 反模式必须在需求拆解阶段就指出
- 验收标准必须包含功能、性能、安全、代码质量、文档五个维度

### 5. 可视化增强理解
- 复杂依赖关系必须绘图
- 架构层次必须可视化
- 业务流程必须有流程图

## 质量标准

### /goal 命令质量标准
1. **完整性**：包含目标、技术栈、架构、约束、分步计划、验收标准
2. **清晰性**：无歧义，AI 和人类都能理解
3. **可执行性**：AI（如 codex）可以基于此 /goal 直接开始实施
4. **可验证性**：验收标准必须可量化、可测试

### 验收标准质量标准
必须包含以下五个维度：

#### 功能完整性
- [ ] 所有 P0 功能点已实现
- [ ] 所有 P1 功能点已实现
- [ ] 功能符合需求描述
- [ ] 边界条件已处理

#### 性能指标
- [ ] 响应时间 < XXXms（根据需求量化）
- [ ] 并发量 > XXX QPS
- [ ] 数据库查询优化（索引、查询计划）
- [ ] 缓存命中率 > XX%（如适用）

#### 安全检查
- [ ] 密码已加密（BCrypt/Argon2）
- [ ] SQL注入防护（参数化查询）
- [ ] XSS防护（输入验证+输出转义）
- [ ] CSRF防护（Token验证）
- [ ] 敏感数据加密传输（HTTPS）
- [ ] 权限验证完整

#### 代码质量
- [ ] 单元测试覆盖率 > 80%
- [ ] 集成测试通过
- [ ] 代码符合团队规范（Linter通过）
- [ ] 无明显技术债务
- [ ] 日志完整（关键操作有日志）

#### 文档完备
- [ ] API文档完整（Swagger/OpenAPI）
- [ ] 部署文档（环境要求、部署步骤）
- [ ] 数据库迁移脚本
- [ ] README（项目说明、快速开始）

## 反模式检测规则

### 架构反模式
1. **业务逻辑在Controller层**
   - 违反：单一职责原则
   - 建议：引入Service层
   
2. **跨服务直接数据库访问**
   - 违反：服务边界
   - 建议：通过API或事件通信

3. **循环依赖**
   - 违反：依赖倒置原则
   - 建议：引入中介者模式或重新设计边界

4. **God Object（上帝对象）**
   - 违反：单一职责原则
   - 建议：拆分为多个内聚的对象

### 安全反模式
1. **密码明文存储**
   - 风险：数据库泄露导致用户账号全部暴露
   - 建议：使用BCrypt/Argon2加密

2. **SQL拼接**
   - 风险：SQL注入攻击
   - 建议：参数化查询或ORM

3. **未验证的用户输入**
   - 风险：XSS、命令注入等
   - 建议：白名单验证+输出转义

4. **硬编码密钥**
   - 风险：代码泄露导致系统被入侵
   - 建议：环境变量或密钥管理服务

### 性能反模式
1. **N+1查询问题**
   - 问题：循环中执行数据库查询
   - 建议：预加载或批量查询

2. **无索引的大表查询**
   - 问题：查询性能低下
   - 建议：添加合适的索引

3. **同步阻塞IO**
   - 问题：高并发时性能瓶颈
   - 建议：异步IO或消息队列

## 拆解粒度标准

### 功能点级别（推荐）
- 到功能点和用户故事
- **不到**具体的类和方法（限制AI的自主创新）

示例：
```
用户登录功能
├── 账号密码验证
├── JWT Token生成
├── Session管理
└── 密码错误次数限制（防暴力破解）
```

### 禁止过度细化
❌ 错误示例：
```
用户登录功能
├── LoginController.login()方法
├── UserService.validatePassword()方法
├── TokenService.generateJWT()方法
```

这会限制AI的实现自由度和创新空间。

## 术语标准化

### DDD术语
- **领域（Domain）**：业务领域，如"电商"、"医疗"
- **限界上下文（Bounded Context）**：领域的边界，如"订单上下文"、"支付上下文"
- **实体（Entity）**：有唯一标识的对象，如"用户"、"订单"
- **值对象（Value Object）**：无唯一标识，只有属性的对象，如"地址"、"价格"
- **聚合根（Aggregate Root）**：聚合的入口实体，如"订单"（包含订单项、支付信息等）
- **领域事件（Domain Event）**：领域中发生的重要事实，如`OrderPlaced`、`PaymentSucceeded`
- **领域服务（Domain Service）**：不属于任何实体的业务逻辑，如"库存分配服务"
- **仓储（Repository）**：数据访问抽象层

### SOA术语
- **服务边界（Service Boundary）**：服务的职责范围
- **服务编排（Orchestration）**：中心化控制，有一个"指挥者"协调所有服务
- **服务协作（Choreography）**：去中心化，服务通过事件自主协作
- **API契约（API Contract）**：接口定义，如OpenAPI/Swagger
- **Saga模式**：分布式事务的解决方案（编排式或协作式）

### 技术术语
- **QPS（Queries Per Second）**：每秒查询数
- **TPS（Transactions Per Second）**：每秒事务数
- **RT（Response Time）**：响应时间
- **P99**：99%的请求响应时间

## 量化标准

### 性能量化
- **高并发**：10000 QPS
- **中等并发**：1000 QPS
- **低并发**：100 QPS
- **快速响应**：< 200ms
- **可接受响应**：< 500ms
- **慢响应**：> 1000ms

### 数据量化
- **大量数据**：百万级以上
- **中等数据**：十万级
- **小量数据**：万级以下

### 可用性量化
- **高可用**：99.9%（年停机时间 < 8.76小时）
- **非常高可用**：99.99%（年停机时间 < 52.6分钟）
- **极高可用**：99.999%（年停机时间 < 5.26分钟）

## 默认假设清单

当用户未明确说明时，基于以下假设（必须在 /goal 中标注）：

### 技术栈假设
- 后端：Java Spring Boot（高并发、企业级）或 Node.js（快速原型）
- 前端：React + TypeScript
- 数据库：PostgreSQL（关系型）+ Redis（缓存）
- 消息队列：Kafka（高吞吐）或 RabbitMQ（可靠性）
- API风格：RESTful + JSON

### 架构假设
- 架构风格：微服务（分布式）或单体（团队<10人）
- 通信方式：异步事件驱动（默认）
- API版本化：默认启用（/api/v1/...）
- 认证方式：JWT Token

### 安全假设
- 密码加密：BCrypt
- HTTPS：启用
- 输入验证：白名单验证
- SQL注入防护：参数化查询/ORM

### 测试假设
- 单元测试：必须有，目标覆盖率 > 80%
- 集成测试：关键流程必须有
- 性能测试：高并发场景必须有
