# 如何构建属于你自己的 OpenGoal Skill

> **一个通过 100+ 问答驱动设计的需求分析专家系统**

---

## 🎯 核心理念

传统的软件开发流程是：**开发者想好需求 → 写代码 → 交付**

但使用 AI 辅助开发时，最大的瓶颈不是编码，而是**如何将模糊的需求转化为 AI 可理解的、结构化的指令**。

OpenGoal Skill 解决的就是这个问题：**从自然语言需求到可执行的 /goal 命令**。

---

## 💡 为什么需要这个 Skill？

### 问题：为什么 AI 在复杂项目中"越用越笨"？

我们分析了 7 个根本原因：

1. **上下文窗口污染** - 有效信息被稀释
2. **目标漂移** - 错误累积效应
3. **抽象层级混乱** - 架构设计和实现细节混淆
4. **依赖关系图缺失** - 无法建立全局视图
5. **上下文一致性丢失** - 早期决策被压缩
6. **缺乏状态检查点** - 无法回溯正确状态
7. **提示词熵增** - 模糊指令引入噪音

### 解决方案：结构化的需求分析流程

OpenGoal Skill 通过以下方式解决这些问题：

- ✅ 结构化需求拆解 → 对抗目标漂移
- ✅ 生成依赖图 → 解决依赖关系缺失
- ✅ 主动提问 → 降低提示词熵增
- ✅ 双版本输出 → 提高信息密度
- ✅ 上下文记忆 → 对抗一致性丢失
- ✅ 验收标准 → 创建状态检查点

## 🏗️ 构建方法：问答驱动设计

### 第一步：明确目标

**核心问题**：你要构建的 Skill 解决什么问题？

对于 OpenGoal，我们的目标是：
> 将自然语言需求转化为结构化的、可执行的 /goal 命令

### 第二步：深度提问（100+ 问题）

这是最关键的一步！**不要急于写代码，先通过大量问题挖掘需求。**

我们将问题分为 10 个维度，每个维度 10 个问题：

#### 1. 基础定位与使用场景（1-10）
- 使用者是谁？
- 输入是什么样的？
- 什么时候使用？
- 支持哪些领域？
- 输出用于什么目的？

#### 2. DDD 领域驱动设计（11-20）
- 如何识别领域？
- 是否需要识别限界上下文？
- 如何处理跨领域需求？
- 是否需要识别实体和值对象？
- 如何识别领域事件？

#### 3. SOA 面向服务架构（21-30）
- 是否需要识别服务边界？
- 如何判断服务拆分？
- 如何识别服务依赖？
- 编排还是协作？
- 如何处理 API 契约？

#### 4. 需求拆解与结构化（31-40）
- 按什么维度拆解？
- 拆解的粒度多细？
- 如何识别功能性和非功能性需求？
- 如何处理依赖关系？
- 如何识别优先级？

#### 5. 提示词生成与优化（41-50）
- /goal 命令包含哪些要素？
- 长度有限制吗？
- 是否需要示例代码？
- 如何处理歧义性？
- 是否需要测试策略？

#### 6-10. 深入细节（51-100）
- 架构风格选择
- 领域事件识别
- 业务规则表达
- 反模式检测
- 输出格式设计
- 质量保证标准
- ... 等等

**关键：每个问题都提供选项，让用户选择，而不是开放式问答。**

### 第三步：整理答案，形成设计决策

将 100 个问题的答案整理成：

1. **核心决策表** - 关键选择和原因
2. **工作契约** - 质量标准和原则
3. **知识库** - 领域知识（如 DDD/SOA）
4. **架构设计** - 模块划分和接口

### 第四步：实现核心功能

基于设计决策，实现以下模块：

1. **需求分析引擎** - 语义分析、实体识别
2. **拆解器** - 按设计的维度拆解需求
3. **决策引擎** - 架构模式选择、技术栈推断
4. **质量检查器** - 反模式识别、风险检测
5. **输出生成器** - 生成 /goal 命令和验收标准
6. **可视化** - 绘制架构图、依赖图等

### 第五步：测试与优化

1. 编写测试用例
2. 用真实需求测试
3. 根据反馈优化
4. 完善文档

## 🎨 OpenGoal Skill 的构建过程

### 1. 问答阶段（实际执行）

我们提出了 **100 个问题**，涵盖：
- 基础定位（10个）
- DDD 概念（10个）
- SOA 概念（10个）
- 需求拆解（10个）
- 提示词生成（10个）
- 架构风格（10个）
- 业务规则（10个）
- 输出格式（10个）
- 质量保证（10个）
- 特殊场景（10个）

**所有问题和答案都记录在 `answer.md` 中。**

### 2. 设计决策（基于答案）

从 100 个答案中，我们提取了关键决策：

| 决策维度 | 选择 | 原因 |
|---------|------|------|
| 架构风格 | 异步事件驱动 | 松耦合、高扩展性 |
| 拆解模式 | 混合模式（功能+技术+领域） | 全面覆盖 |
| 输出格式 | 双版本（人类+AI） | 各自优化 |
| 验收标准 | 5个维度 | 全面质量保证 |
| 绘图方式 | matplotlib + networkx | 继承 nature-figure |

### 3. 知识库建设

基于"不懂"的问题，我们进行了联网研究，建立了完整的知识库：

- **DDD 知识库**：8个核心概念，每个都有定义、识别方法、示例
- **SOA 知识库**：7个核心概念，每个都有适用场景、优缺点、案例
- **架构决策树**：何时同步 vs 异步、何时编排 vs 协作

### 4. 功能实现

基于设计决策，我们实现了 9 步工作流程：

1. 接收用户输入
2. 语义分析
3. 拆解需求
4. 识别依赖关系
5. 识别反模式
6. 生成补充问题
7. 整合答案
8. 生成 /goal 命令
9. 输出完整交付物

### 5. 测试与文档

- 编写了 2 个测试套件（完整版和简化版）
- 编写了 5 个文档（README、SUMMARY、DELIVERY 等）
- 创建了示例和使用指南

## 📝 构建自己的 Skill：具体步骤

### 步骤 1：定义领域

选择一个你熟悉的领域，明确要解决的问题。

**示例**：我要构建一个"API 设计专家 Skill"

### 步骤 2：设计问题

参考 OpenGoal 的结构，设计 10 个维度的问题：

1. **基础定位**（10个）
   - 支持哪些 API 风格？（REST/GraphQL/gRPC）
   - 输入是什么？（需求文档/用户故事）
   - 输出是什么？（OpenAPI/Swagger/文档）

2. **API 设计原则**（10个）
   - RESTful 规范遵守程度？
   - 如何设计资源路径？
   - 如何设计查询参数？

3. **版本管理**（10个）
   - 版本化策略？（URI/Header/Query）
   - 向后兼容原则？
   - 破坏性变更处理？

4. **安全设计**（10个）
   - 认证方式？（JWT/OAuth2/API Key）
   - 授权模式？（RBAC/ABAC）
   - 限流策略？

5. **错误处理**（10个）
   - 错误码设计？（HTTP状态码/自定义码）
   - 错误消息格式？
   - 如何处理验证错误？

... 继续到 100+ 问题

### 步骤 3：与 AI 对话

将这些问题一个一个地问 AI（如 Claude），让 AI 帮你回答并记录。

**技巧**：
- 每个问题都提供选项（A/B/C/D）
- 不确定的问题标注"不懂，请联网研究"
- 让 AI 帮你补充更多问题

### 步骤 4：整理答案

将所有答案整理成设计决策表：

```markdown
| 问题 | 答案 | 原因 |
|------|------|------|
| API 风格 | REST优先 | 通用性最好 |
| 版本化 | URI版本 | 最直观 |
| 认证 | JWT | 无状态、易扩展 |
```

### 步骤 5：让 AI 帮你实现

基于整理好的答案，让 AI 帮你生成：

1. **Skill 主文件** - 定义功能和工作流程
2. **知识库** - API 设计最佳实践
3. **工作契约** - 质量标准和反模式
4. **执行脚本** - 核心逻辑实现

### 步骤 6：测试与优化

用真实案例测试你的 Skill，根据反馈优化。

## 💎 关键成功因素

### 1. 问题质量 > 问题数量

100 个深思熟虑的问题 > 1000 个随意的问题

**好问题的特点**：
- 有明确的选项
- 覆盖关键决策点
- 能引导深度思考

### 2. 领域知识积累

对于"不懂"的问题，不要猜测，要：
- 联网研究
- 查阅文档
- 学习最佳实践

### 3. 迭代优化

第一版不需要完美，但要：
- 核心功能完整
- 文档清晰
- 可测试

然后根据使用反馈持续优化。

### 4. 文档驱动

好的 Skill = 好的代码 + 好的文档

**必需的文档**：
- README.md - 使用指南
- 知识库 - 领域知识
- 工作契约 - 质量标准
- 示例 - 实际案例
