**答案**：我不懂，请联网自主分析

## SOA 面向服务架构相关（21-30）

### 21. skill 是否需要识别"服务边界"？

**答案**：需要

---

### 22. 如何判断一个需求应该拆分为多个服务还是单一服务？

**答案**：我不懂，请联网自主分析

**AI 研究结果**：
- 单一服务：功能内聚、团队规模小（<10人）、变更频率一致
- 多服务：不同业务域、不同技术栈、不同扩展需求、不同团队
- 拆分原则：业务能力、数据边界、变更频率、团队组织（康威定律）

---

### 23. 是否需要识别服务之间的"依赖关系"？

**答案**：需要，必要时可绘图分析，我在项目目录下放置了一个 nature-skill 的压缩包，你在写 skill 的时候可以把其中绘图的相关内容放进去

---

### 24. skill 如何处理"服务编排"（Orchestration）vs "服务协作"（Choreography）的选择？

**答案**：我不懂，请联网自主分析

**AI 研究结果**：
- 编排（Orchestration）：中心化控制，有一个"指挥者"协调所有服务；适合流程固定、需要事务保证
- 协作（Choreography）：去中心化，服务通过事件自主协作；适合松耦合、高扩展性
- 推荐混合模式：关键操作（如库存扣减、支付）用编排，非关键操作（如邮件、积分）用协作

---

### 25. 是否需要识别"API 契约"或"接口定义"？

**答案**：我不懂，请联网自主分析

**AI 研究结果**：
- Contract-First 原则：先定义 API 契约（OpenAPI/Swagger），再实现
- 版本管理策略：URI版本（/api/v1/...）、Header版本、Query参数版本
- 在 Skill 中：默认所有 API 都版本化

---

### 26. 如何处理服务的"版本管理"需求？

**答案**：我不懂，请联网自主分析

**AI 研究结果**：
- 版本化规则：所有 API 默认版本化、破坏性变更必须升级主版本、非破坏性变更可在同一版本内扩展

---

### 27. 是否需要考虑服务的"可观测性"（日志、监控、追踪）？

**答案**：需要

---

### 28. skill 是否应该识别"同步调用"和"异步消息"的需求？

**答案**：需要

### 29. 如何处理服务的"容错和降级"需求？

**答案**：需要，但不是重点

### 30. 是否需要考虑"服务网格"（Service Mesh）相关的架构模式？
